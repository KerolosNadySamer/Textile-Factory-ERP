# قائمة اختبارات التشغيل التجريبي النهائية

هذه القائمة مخصصة لمرحلة ما قبل التشغيل الفعلي داخل المصنع. الهدف ليس إضافة وظائف جديدة، بل التأكد أن النظام يعمل بسلاسة مع المستخدمين الحقيقيين، الصلاحيات، الموافقات، السجلات، الإشعارات، والأداء.

## قواعد التنفيذ

- نفذ الاختبارات على بيانات تجريبية واضحة، ولا تستخدم بيانات تشغيل حقيقية قبل اعتماد النتيجة.
- سجل نتيجة كل اختبار: `ناجح` أو `فشل` أو `يحتاج تعديل`.
- احتفظ بلقطة شاشة أو رقم مستند لكل نتيجة مهمة مثل Change Request أو Sales Order أو Purchase Order أو Backup.
- أي فشل في الصلاحيات أو الموافقات أو النسخ الاحتياطي يعتبر مانع تشغيل ويجب إصلاحه قبل التجربة الفعلية.

## 1. اختبار الصلاحيات

الأدوار المطلوب اختبارها:

- Admin
- General Manager
- Sales Manager
- Sales User
- Purchasing Manager
- Purchasing User
- Warehouse
- Production
- Quality
- Cost Accountant

خطوات الاختبار:

1. سجل الدخول بكل دور.
2. راجع عناصر القائمة الرئيسية.
3. جرّب فتح صفحات لا تخص الدور مباشرة من الرابط إن أمكن.
4. جرّب إنشاء أو تعديل أو حذف سجل حسب صلاحيات الدور.

النتيجة المتوقعة:

- كل مستخدم يرى فقط الصفحات والعمليات المسموح بها.
- أي صفحة غير مصرح بها تعطي منع وصول.
- المستخدم غير المصرح لا يستطيع تنفيذ العملية حتى لو عرف الرابط.

دليل الإثبات:

- اسم المستخدم والدور.
- الصفحات الظاهرة.
- أي رسالة منع وصول.

## 2. اختبار طبقة الموافقات

السيناريوهات:

- إنشاء عميل
- تعديل عميل
- تعديل منتج
- تعديل سعر
- تعديل مستخدم

خطوات الاختبار:

1. نفذ العملية من مستخدم لا يملك تنفيذًا مباشرًا.
2. تأكد من إنشاء Change Request.
3. وافق عليها من المستخدم المسؤول.
4. تأكد من تنفيذ التغيير على السجل الأصلي.
5. راجع Audit Log.
6. راجع Notification للمستخدمين المعنيين.

النتيجة المتوقعة:

- Change Request يتم إنشاؤه بالبيانات الصحيحة.
- Approval ينقل الطلب للحالة الصحيحة.
- Execution يحدث التغيير فعليًا.
- Audit Log يسجل العملية.
- Notification تصل للطرف المناسب فقط.

دليل الإثبات:

- رقم Change Request.
- اسم الموافق.
- وقت التنفيذ.
- سجل Audit Log.

## 3. اختبار Switch User

خطوات الاختبار:

1. سجل الدخول بالمستخدم A.
2. افتح قائمة المستخدم واختر تغيير المستخدم.
3. اختر المستخدم B.
4. أدخل كلمة مرور المستخدم B.
5. تأكد أن الجلسة تحولت للمستخدم B.
6. افتح User Switch History.
7. راجع Activity Logs.

النتيجة المتوقعة:

- لا يتم التبديل بدون كلمة مرور المستخدم الجديد.
- بعد التبديل تظهر صلاحيات المستخدم الجديد فقط.
- يتم تسجيل From وTo وTime وIP وDevice.
- آخر 10 مستخدمين على الجهاز تظهر بشكل صحيح.

دليل الإثبات:

- From User.
- To User.
- وقت التبديل.
- IP.

## 4. اختبار الاستيراد

ملف الاختبار يجب أن يحتوي:

- بيانات صحيحة.
- بيانات مكررة.
- بيانات ناقصة.

خطوات الاختبار:

1. حمل نموذج استيراد العملاء.
2. جهز ملف يحتوي الحالات الثلاث.
3. ارفع الملف.
4. راجع شاشة كشف التكرارات.
5. جرّب اختيار إدخال البيانات غير المكررة فقط.
6. جرّب قبول التكرارات إذا كان السيناريو يسمح.

النتيجة المتوقعة:

- البيانات الصحيحة تدخل بنجاح.
- البيانات المكررة تظهر في كشف واضح.
- البيانات الناقصة لا تدخل أو تظهر برسالة خطأ واضحة.
- لا تضيع البيانات المعلقة أثناء قرار التكرارات.

دليل الإثبات:

- اسم ملف الاستيراد.
- عدد الصفوف الصحيحة.
- عدد الصفوف المكررة.
- عدد الصفوف المرفوضة.

## 5. اختبار دورة المشتريات كاملة

التسلسل:

Supplier -> Purchase Request -> Purchase Order -> Goods Receipt -> Lot -> Inventory Ledger

خطوات الاختبار:

1. أنشئ موردًا.
2. أنشئ Purchase Request.
3. حوّل الطلب إلى Purchase Order.
4. اعتمد Purchase Order.
5. أنشئ Goods Receipt.
6. تأكد من إنشاء أو تحديث Lot.
7. راجع Inventory Ledger.

النتيجة المتوقعة:

- كل خطوة مرتبطة بما قبلها.
- الكميات والأسعار والوحدات تنتقل بشكل صحيح.
- المخزون يظهر في Ledger بعد الاستلام.
- الإشعارات تصل للمشتريات والمخازن والإدارة حسب الحالة.

دليل الإثبات:

- رقم المورد.
- رقم PR.
- رقم PO.
- رقم GRN.
- رقم Lot.
- حركة Inventory Ledger.

## 6. اختبار دورة المبيعات

التسلسل:

Customer -> Sales Order -> Approvals -> Production Order

خطوات الاختبار:

1. أنشئ أو اختر عميلًا.
2. أنشئ Sales Order.
3. نفذ دورة الاعتماد.
4. أنشئ أو راجع Production Order الناتج.

النتيجة المتوقعة:

- Sales Order يحتوي بيانات العميل والأصناف والكميات.
- الموافقات تعمل حسب الصلاحيات.
- Production Order مرتبط بطلب البيع.
- إشعارات التخطيط والإنتاج تظهر في الوقت المناسب.

دليل الإثبات:

- رقم العميل.
- رقم Sales Order.
- حالة الاعتماد.
- رقم Production Order.

## 7. اختبار التتبع

نقطة البداية:

- عميل

المطلوب الوصول إليه:

- Sales Order
- Production Order
- Lot
- Supplier

خطوات الاختبار:

1. اختر عميلًا له دورة مكتملة.
2. راجع طلب البيع المرتبط.
3. راجع أمر الإنتاج المرتبط.
4. راجع اللوطات المرتبطة.
5. راجع المورد أو مصدر الخامة إن وجد.

النتيجة المتوقعة:

- تقرير التتبع يعرض سلسلة العلاقة كاملة.
- لا توجد حلقات مفقودة في السلسلة.
- أرقام المستندات متطابقة مع السجلات الأصلية.

دليل الإثبات:

- اسم العميل.
- أرقام المستندات في السلسلة.
- لقطة من تقرير التتبع.

## 8. اختبار النسخ الاحتياطي والاسترجاع

خطوات الاختبار:

1. نفذ Backup.
2. سجل رقم أو اسم ملف النسخة.
3. أضف بيانات اختبار واضحة.
4. نفذ Restore للنسخة.
5. تأكد أن بيانات الاختبار التي أضيفت بعد النسخة اختفت.
6. تأكد أن بيانات ما قبل النسخة موجودة.

النتيجة المتوقعة:

- النسخة يتم إنشاؤها بنجاح.
- الاسترجاع يعيد الحالة الأصلية.
- سجل النسخ يظهر من أنشأ النسخة ومن استرجعها.

دليل الإثبات:

- اسم ملف Backup.
- وقت الإنشاء.
- وقت الاسترجاع.
- السجل الذي اختفى بعد الاسترجاع.

## 9. اختبار الإشعارات

الأقسام:

- Sales
- Planning
- Warehouse
- Production
- Quality
- Management

خطوات الاختبار:

1. نفذ أحداثًا تولد إشعارات لكل قسم.
2. سجل الدخول بكل مستخدم معني.
3. راجع عداد الإشعارات والقائمة.
4. علّم إشعارًا كمقروء.

النتيجة المتوقعة:

- الإشعار يصل للمستخدم أو القسم المناسب فقط.
- غير المعنيين لا يظهر لهم الإشعار.
- عداد الإشعارات يتغير بعد القراءة.

دليل الإثبات:

- الحدث.
- المستلمون.
- المستخدمون الذين لم يصلهم الإشعار.

## 10. اختبار الأداء

أحجام البيانات:

- 100 عميل
- 1000 عميل
- 5000 عميل

خطوات الاختبار:

1. جهز بيانات العملاء بالأحجام الثلاثة.
2. اختبر فتح قائمة العملاء.
3. اختبر البحث العام.
4. اختبر الفلاتر.
5. اختبر التقارير أو التصدير إن وجد.

النتيجة المتوقعة:

- القوائم تفتح بزمن مقبول.
- البحث لا يتوقف أو يتأخر بشكل واضح.
- الفلاتر تعمل بدقة.
- لا تظهر أخطاء ذاكرة أو timeout.

دليل الإثبات:

- حجم البيانات.
- زمن فتح القائمة.
- زمن البحث.
- ملاحظات المستخدمين.
