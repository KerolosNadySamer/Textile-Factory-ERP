# سياسة الحفاظ على البيانات أثناء التطوير

هذه السياسة قاعدة إلزامية لتطوير نظام الـ ERP بعد بدء إدخال بيانات فعلية أو شبه فعلية. أي تعديل جديد يجب أن يكون Backward Compatible ويحافظ على كل البيانات والحركات والمستندات السابقة.

## القاعدة الأساسية

- لا يتم حذف بيانات موجودة بسبب إضافة ميزة جديدة.
- لا يتم حذف حركات سابقة.
- لا يتم حذف أو تعديل مستندات معتمدة إلا بإجراء واضح ومصرح ومسجل.
- الحقول الجديدة تكون اختيارية `nullable` أو لها قيمة افتراضية آمنة.
- أي Workflow جديد يجب أن يستوعب البيانات القديمة.
- التقارير الجديدة تقرأ من البيانات الحالية ولا تنشئ بيانات بديلة.
- يمنع استخدام `truncate` أو أوامر حذف جماعي على جداول التشغيل بعد بدء التشغيل.

## الجداول المحمية

هذه الجداول تعتبر جداول تشغيل أو سجلات حساسة، ولا يجوز حذف بياناتها أثناء التطوير:

- customers
- suppliers
- sales_orders
- sales_order_items
- production_orders
- production_order_items
- purchase_requests

## قواعد الـ Migrations

استخدم:

```php
Schema::table('table_name', function (Blueprint $table) {
    $table->string('new_column')->nullable();
});
```

وتجنب في migrations الجديدة بعد التشغيل:

```php
Schema::dropIfExists(...);
DB::table(...)->truncate();
DB::table(...)->delete();
```

الاستثناء الوحيد هو `down()` الخاص بالـ migration أثناء التطوير المحلي قبل التشغيل، أو migration تجريبي غير مطبق على بيانات فعلية. بعد التشغيل، يجب التعامل مع `down()` بحذر شديد وعدم تشغيل rollback على بيئة تشغيل بدون Backup واعتماد.

## قواعد تعديل الجداول القديمة

عند إضافة حقول جديدة:

- اجعلها `nullable` أو أعطها default مناسب.
- لا تجعل الحقل الجديد `required` على البيانات القديمة فورًا.
- أضف Backfill تدريجي إذا احتاج التقرير أو workflow قيمة محددة.
- لا تعيد إنشاء السجلات القديمة.

مثال مستخدمين:

```text
إضافة صورة شخصية / مدير مباشر / قسم / وظيفة
↓
المستخدمون الحاليون يظلون كما هم
↓
القيم الجديدة تكون Null
↓
يتم استكمالها لاحقًا
```

مثال عملاء:

```text
إضافة مندوب مبيعات / حد ائتماني / تصنيف
↓
العملاء الحاليون يظلون موجودين
↓
الحقول الجديدة Null أو Default
↓
استكمال تدريجي بدون إعادة إنشاء العملاء
```

## قواعد الـ Seeders

الـ seeders ليست أداة تحديث بيانات تشغيل.

بعد بدء التشغيل:

- لا تستخدم `migrate:fresh`.
- لا تستخدم seeder يعيد ضبط كلمات المرور أو الأدوار أو البيانات المدخلة.
- إذا كان لا بد من seeder لبيانات مرجعية، استخدم `firstOrCreate` أو `updateOrInsert` بحذر.
- لا تغير سجلًا موجودًا إلا إذا كان هذا التغيير مقصودًا ومذكورًا في خطة الإصدار.

للحسابات التجريبية:

- يتم إنشاؤها فقط إذا لم تكن موجودة.
- لا يتم إعادة ضبط كلمة المرور إذا كان الحساب موجودًا.
- لا يتم تغيير بيانات الحساب إذا قام المستخدم بتعديلها لاحقًا.

## قواعد الـ Workflows الجديدة

عند إضافة Approval Layer أو Workflow جديد:

- البيانات القديمة تعتبر Approved أو Executed حسب طبيعتها.
- البيانات الجديدة فقط تدخل في workflow الجديد.
- لا يتم تعليق أو تعطيل المستندات القديمة لأن workflow أضيف لاحقًا.

مثال:

```text
قبل إضافة Approval Layer:
كل العملاء والطلبات الحالية = Approved

بعد إضافة Approval Layer:
الإنشاءات والتعديلات الجديدة فقط = Pending Approval
```

## قواعد الحذف داخل النظام

يفضل الحذف المنطقي أو الإلغاء بدل الحذف النهائي:

- المستند المعتمد لا يحذف.
- المستند المرتبط بحركة مخزون لا يحذف.
- اللوط المرتبط بتتبع أو حركة مخزون لا يحذف.
- المستخدم الذي أنشأ حركات لا يحذف حذفًا نهائيًا، بل يتم تعطيل الدخول.

الحذف المسموح:

- بيانات إعداد غير مستخدمة.
- سجلات تجريبية قبل بدء التشغيل.
- مرفقات أو ملفات مؤقتة غير مرتبطة بمستندات تشغيل.

## النسخ الاحتياطي والاسترجاع

ميزة Restore بطبيعتها تستبدل الحالة الحالية ببيانات النسخة الاحتياطية، لذلك:

- لا يتم استخدامها في التشغيل إلا بقرار إداري واضح.
- يجب أخذ Backup جديد قبل أي Restore.
- يجب تسجيل من نفذ الاسترجاع ووقته.
- Restore يستخدم لإنقاذ النظام، وليس كطريقة تحديث أو تطوير.

## مراجعة قبل أي تعديل جديد

قبل تنفيذ أي إضافة أو تعديل، أجب عن الأسئلة التالية:

- هل هذا التعديل يحذف أي بيانات؟
- هل يغير حالة مستندات معتمدة؟
- هل يجعل حقلًا جديدًا إجباريًا على سجلات قديمة؟
- هل يحتاج Backfill آمن؟
- هل سيؤثر على التقارير الحالية؟
- هل يمكن تشغيله بدون `migrate:fresh`؟
- هل يمكن الرجوع منه بدون فقد بيانات؟

إذا كانت الإجابة على أي سؤال تحمل خطرًا، يجب تنفيذ Backup وخطة Migration واضحة قبل التعديل.

## قاعدة مختصرة للمطورين

```text
أي تعديل جديد يجب أن يكون Backward Compatible.

- لا حذف لبيانات موجودة.
- لا حذف لحركات سابقة.
- لا حذف لمستندات معتمدة.
- الحقول الجديدة اختيارية أو لها Default.
- أي Workflow جديد يستوعب البيانات القديمة.
- ممنوع truncate أو drop على جداول التشغيل.
- Seeders لا تعيد ضبط بيانات مستخدمة.
```
